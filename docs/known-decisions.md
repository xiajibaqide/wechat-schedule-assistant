# 已知项目决策

本文记录微信日程助手当前已经形成的设计决策。后续开发如果要改变这些决策，应先明确原因、影响范围和兼容方案。

## 1. 项目保持前端-only

### 决策

当前项目只使用 React + Vite + JavaScript 在浏览器中运行。

### 原因

- 项目目标是本地运行的 AI 编程练习和 MVP 原型。
- 当前核心价值是验证消息输入、日程提取、人工确认和本地管理流程。
- 增加后端会扩大复杂度，不利于小步迭代。

### 影响

- 不提供服务器 API。
- 不提供数据库。
- 不提供用户登录。
- 不提供多端同步。
- 页面关闭后，自动提醒循环会停止。

### 后续注意事项

- 不要在当前阶段引入后端或数据库。
- 如果未来确实需要后端，应先单独设计数据模型、同步策略和迁移方案。

## 2. 不接入真实微信

### 决策

当前项目只模拟微信群消息输入，不接入真实微信接口。

### 原因

- 真实微信接入涉及平台能力、合规要求和权限问题。
- 当前 MVP 只需要验证“从群消息文本提取日程”的产品流程。

### 影响

- 用户需要手动输入或选择示例消息。
- 项目不读取真实微信聊天记录。
- 项目不发送微信提醒。

### 后续注意事项

- 文档和 UI 不应暗示项目已经接入真实微信。
- 未来评估真实微信接入时，应先处理合规、授权和隐私边界。

## 3. 不调用真实大模型 API

### 决策

当前提取逻辑使用本地规则引擎，不调用真实大模型 API。

### 原因

- 规则引擎更适合当前 MVP 的本地、可解释、低成本目标。
- 不需要配置 API key。
- 不需要处理网络失败、费用和模型输出不稳定问题。

### 影响

- 复杂自然语言理解能力有限。
- 日期、时间、地点和标题识别依赖规则覆盖。
- 置信度是规则评分，不代表模型概率。

### 后续注意事项

- 如果未来加入大模型辅助，应保留人工确认流程。
- 大模型输出也应先变成草稿，不能直接写入 confirmed 日程。
- `hybridExtractor.js` 可以作为未来扩展入口，但当前仍以 rules 为准。

## 4. 用户确认是核心流程

### 决策

系统提取出的日程先进入待确认草稿，由用户确认后才保存为本地日程。

### 原因

- 群聊文本容易包含普通聊天、玩笑或不完整信息。
- 规则提取可能误判日期、时间、地点或标题。
- 人工确认可以降低误保存风险。

### 影响

- `draftEvent` 不会自动进入本地日程列表。
- 用户确认时，事件 status 才会变成 `confirmed`。
- 用户忽略的草稿当前不会持久化到 localStorage。

### 后续注意事项

- 新增提取能力时，不应绕过确认步骤。
- 低置信度草稿仍应允许用户查看和判断。

## 5. 数据保存在 localStorage

### 决策

已保存日程使用浏览器 localStorage 持久化。

当前 key：

```text
wechat-schedule-assistant-events
```

### 原因

- 符合前端-only MVP 的边界。
- 实现简单，便于本地演示。
- 不需要后端、数据库或登录系统。

### 影响

- 数据只保存在当前浏览器环境。
- 清除浏览器数据后，日程会丢失。
- 不支持跨设备同步。

### 后续注意事项

- 不要随意更改 localStorage key。
- 修改事件字段时应兼容旧数据。
- 保存的数据必须保持 JSON-compatible。
- 读取失败时应安全回退为空数组。

## 6. status 只表示用户决策

### 决策

事件 status 只允许以下值：

```text
pending
confirmed
dismissed
```

### 原因

- status 用来表达用户对提取结果的处理结果。
- 提醒、结束等流程属于生命周期，不属于用户决策。
- 保持 status 简洁可以减少状态组合混乱。

### 影响

- `pending` 表示待确认。
- `confirmed` 表示用户确认保存。
- `dismissed` 表示用户忽略。
- 不使用 `reminded` 或 `ended` 作为 status。

### 后续注意事项

- 不要向 status 枚举新增 `reminded`、`ended` 等生命周期状态。
- 新增 UI 流程步骤时，应与 status 保持解耦。

## 7. 生命周期使用时间戳字段

### 决策

提醒和结束状态使用时间戳字段记录：

```text
remindedAt
endedAt
```

### 原因

- 生命周期可以和用户决策状态同时存在。
- 时间戳能记录事件发生时间，比布尔值信息更完整。
- 可选字段对旧 localStorage 数据更友好。

### 影响

- 一个事件可以同时是 `status: 'confirmed'` 且拥有 `remindedAt`。
- 一个事件可以同时是 `status: 'confirmed'` 且拥有 `endedAt`。
- 工作流统计通过 helper 判断生命周期字段。

### 后续注意事项

- 写入 `remindedAt` 和 `endedAt` 时，应通过 React state setter 做不可变更新。
- 旧事件缺少这些字段时，应视为尚未提醒或尚未结束。

## 8. 提醒设置使用 reminderMinutes

### 决策

每个事件使用 `reminderMinutes` 表示提前多少分钟提醒。

### 原因

- 数值字段简单、可持久化、易于和时间计算结合。
- `0` 可以明确表示不提醒。
- 不需要为提醒设置新增复杂对象。

### 影响

- 默认提醒提前量是 10 分钟。
- `0` 表示不提醒。
- 支持 5、10、30、60 分钟等预设值。

### 后续注意事项

- 旧事件可能没有 `reminderMinutes`，读取时应使用默认值。
- 不要把“不提醒”写成新的 status。
- 如果未来支持自定义分钟数，应增加数值校验。

## 9. 事件持续时间使用 durationMinutes

### 决策

每个事件使用 `durationMinutes` 表示持续时间。

### 原因

- 持续时间需要被提醒结束判断和 ICS 导出共同使用。
- 数值字段便于计算结束时间。
- 可选字段能兼容旧事件。

### 影响

- 默认持续时间是 60 分钟。
- 用户可以在确认和编辑阶段设置持续时间。
- ICS 导出使用 `durationMinutes` 生成有时间事件的 DTEND。
- 事件结束判断也基于该字段。

### 后续注意事项

- 旧事件可能没有 `durationMinutes`，读取时应使用默认值。
- 修改时长选项时，应同步检查确认表单、编辑表单、结束判断和 ICS 导出。
- 全天事件仍应按日期型 DTSTART/DTEND 处理。

## 10. ICS 导出只导出可落地日程

### 决策

ICS 导出只处理 confirmed 且有日期的事件。

### 原因

- 未确认事件不应进入外部日历。
- 没有日期的事件无法形成有效日历日期。
- 保持导出结果可被常见日历应用导入。

### 影响

- confirmed 且有日期的事件可以导出。
- 有时间的事件导出开始和结束时间。
- 无时间但有日期的事件按全天事件导出。

### 后续注意事项

- 不要导出 pending 草稿。
- 不要导出 dismissed 事件。
- 修改导出字段时，要继续处理中文标题、地点和描述的转义。

## 11. 浏览器通知是可选能力

### 决策

浏览器通知用于本地提醒，但通知权限失败不能阻断应用。

### 原因

- 用户可能拒绝通知权限。
- 某些浏览器或环境可能不支持通知。
- 提取、确认和保存日程是更核心的流程。

### 影响

- 应用启动时可以请求通知权限。
- 请求失败时不影响主界面使用。
- 测试通知和自动提醒都依赖浏览器权限。

### 后续注意事项

- 不要让通知失败导致页面报错或流程中断。
- 自动提醒只能保证页面打开且浏览器允许通知时工作。

## 12. 保持小步迭代和 beginner-friendly 结构

### 决策

项目开发优先小范围修改，保持代码结构直接、易读。

### 原因

- 项目是学习和演示用途。
- 清晰的数据流比过早抽象更重要。
- 小步迭代更容易检查兼容性和回归风险。

### 影响

- 优先沿用现有组件和工具模块边界。
- 不为单个小功能引入过重抽象。
- 不重构无关文件。

### 后续注意事项

- 新增功能前先确认现有数据流。
- 修改共享工具函数时，应检查所有调用方。
- 文档、测试清单和代码应随着关键行为变化逐步更新。
