# 项目版本历史

本文记录微信日程助手从 MVP 到当前开发版本的主要演进。内容以 Git 标签、当前分支和项目现有实现为依据，用于后续开发时快速理解每个版本的边界和注意事项。

## v1.0-mvp

### 版本状态

- 已完成。
- 已作为早期 MVP 标签保留。

### 完成内容

- 建立前端单页面应用基础。
- 支持模拟微信群消息输入。
- 支持从文本中提取日程草稿。
- 支持用户确认日程并保存到本地。
- 支持本地日程列表展示、编辑和删除。
- 优化普通聊天误识别，降低非日程消息的置信度。

### 主要功能变化

- 从“手动输入日程”推进到“从消息中提取日程草稿”。
- 提取字段包括标题、日期、时间、地点、原始消息、状态和置信度。
- 普通聊天即使包含日期词，也不会被强行当作高置信度日程。

### 主要设计变化

- 项目确定为前端-only MVP。
- 数据保存在浏览器 localStorage。
- 提取逻辑采用本地规则引擎，不接入真实微信、后端服务或大模型 API。
- 用户确认仍是核心流程，系统只生成待确认草稿。

### 后续注意事项

- 规则提取能力有限，复杂中文日期表达仍需逐步补充。
- 低置信度草稿需要继续依赖用户判断。
- localStorage 数据结构后续修改时应保持兼容。

## v1.3-notification

### 版本状态

- 已完成。
- 已作为浏览器通知能力版本标签保留。

### 完成内容

- 新增浏览器通知服务。
- 应用启动时请求浏览器通知权限。
- 在本地日程列表中加入测试通知入口。

### 主要功能变化

- 项目开始具备浏览器通知能力。
- 用户可以手动触发测试通知，验证浏览器权限和通知展示是否可用。

### 主要设计变化

- 通知能力封装到独立工具模块中。
- 通知权限被视为可选能力，请求失败不应阻断主流程。
- 通知功能仍然运行在浏览器本地，不依赖后端推送。

### 后续注意事项

- 浏览器通知依赖用户授权和浏览器支持。
- 通知失败不能影响提取、确认、保存等核心流程。
- 后续自动提醒应复用通知服务，而不是在组件中重复实现通知逻辑。

## v1.3b-reminder

### 版本状态

- 已完成。
- 已作为自动提醒能力版本标签保留。

### 完成内容

- 新增自动提醒服务。
- 应用启动后定时检查即将开始的 confirmed 日程。
- 对符合提醒条件的事件触发浏览器通知。

### 主要功能变化

- 从“手动测试通知”扩展为“自动检查并提醒”。
- 日程提醒开始与事件日期和时间关联。
- 已确认日程成为提醒服务的主要处理对象。

### 主要设计变化

- 提醒检查逻辑从 UI 组件中分离到工具模块。
- App 负责启动和停止提醒循环。
- 提醒功能通过事件提供函数读取最新 React 状态，避免依赖过期闭包。

### 后续注意事项

- 自动提醒只在页面打开时运行。
- 无日期或无时间的事件不适合自动定时提醒。
- 提醒循环频率和重复提醒控制需要谨慎维护。

## v1.4-reminder-settings

### 版本状态

- 已完成。
- 已作为提醒设置版本标签保留。

### 完成内容

- 新增日程提醒提前量设置。
- 确认日程时可以选择不提醒、提前 5 分钟、10 分钟、30 分钟或 1 小时。
- 编辑已保存日程时可以调整提醒设置。
- 为旧数据提供默认提醒提前量。
- 补充提醒设置相关手动测试内容。

### 主要功能变化

- 提醒从固定规则变为用户可配置。
- 每个事件可以保存自己的 reminderMinutes。
- 旧事件没有 reminderMinutes 时，默认按 10 分钟处理。

### 主要设计变化

- reminderMinutes 成为事件对象的可选持久化字段。
- 默认值通过工作流/提醒相关工具函数统一处理。
- “不提醒”使用数值 0 表示，而不是新增事件状态。

### 后续注意事项

- 修改提醒逻辑时要继续兼容缺少 reminderMinutes 的旧 localStorage 数据。
- 不能把“不提醒”设计成新的 status。
- 手动测试清单当前在终端中可能出现编码显示问题，后续编辑时应注意文件编码和中文显示。

## v1.5-workflow

### 版本状态

- 已完成。
- 当前正式版本。
- 已打标签 v1.5-workflow。

### 完成内容

- 新增工作流辅助模块。
- 展示原始消息、已提取、待确认、已确认、已提醒、已结束等流程信息。
- 通过 App state 记录 remindedAt 和 endedAt。
- 在本地日程列表中展示生命周期状态。

### 主要功能变化

- 用户可以看到日程从提取到确认，再到提醒和结束的流程状态。
- 已提醒和已结束不再只是临时 UI 判断，而是记录到事件数据中。
- 日程结束判断开始基于事件开始时间和持续时间规则。

### 主要设计变化

- 事件 status 仍然只表示用户决策：pending、confirmed、dismissed。
- remindedAt 和 endedAt 是生命周期时间戳，不是 status。
- 工作流展示步骤与事件 status 解耦，允许 UI 展示 broader workflow。
- 状态更新保持在 React state setter 内完成，避免直接修改事件对象。

### 后续注意事项

- 不要向 status 枚举新增 reminded、ended 等生命周期状态。
- 后续功能如果需要记录生命周期，应优先增加可选时间戳字段。
- 修改工作流计数时，需要同时考虑草稿事件和已保存事件。

## v1.6-event-duration（开发中）

### 版本状态

- 开发中。
- 当前分支为 codex/v1.6-event-duration。
- 尚未作为正式版本标签发布。

### 完成内容

- 新增事件持续时间基础字段。
- 确认日程时保存 durationMinutes。
- 编辑已保存日程时可以调整持续时间。
- ICS 导出使用 durationMinutes 生成 DTEND。
- 旧数据没有 durationMinutes 时默认按 60 分钟处理。

### 主要功能变化

- 事件不再固定视为 1 小时。
- 用户可以在确认和编辑阶段选择 30、60、90 或 120 分钟。
- 导出的 .ics 文件会根据事件持续时间生成结束时间。
- 全天事件仍按日期型 DTSTART/DTEND 导出。

### 主要设计变化

- durationMinutes 成为事件对象的可选持久化字段。
- 默认持续时间集中为 60 分钟。
- ICS 导出通过工作流工具函数读取持续时间，保持旧数据兼容。
- 事件结束判断与持续时间字段保持一致。

### 后续注意事项

- 发布前应继续验证构建和 ICS 导入结果。
- 修改 durationMinutes 时要兼容没有该字段的旧 localStorage 事件。
- 需要避免把“已结束”写入 status，仍应使用 endedAt。
- 后续如增加自定义时长输入，应保持表单和导出逻辑的数值校验。
